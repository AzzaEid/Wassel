# وصّل — السيناريو النهائي المؤكد
## مبني على قرارات الفريق خطوة بخطوة

---

## القرارات المثبّتة

| الموضوع | القرار |
|---------|--------|
| إنشاء الطلب — طريقتان | **يدوي (source=manual):** التاجر ينشئ على موقع وصّل مباشرة — لا شركة توصيل. **تلقائي (source=webhook):** التاجر يسجّل طرداً عند شركة التوصيل، تُرسل Webhook تلقائياً لوصّل |
| بيانات الطلب | هاتف + اسم الزبون + منتج + مبلغ (₪ ثابت) + عنوان |
| اسم الزبون (يدوي) | حقل إلزامي في Dashboard عند إنشاء الطلب يدوياً — لا يُملأ تلقائياً للزبائن الجدد |
| Risk Score | يظهر بعد زر "تحقق" — القرار للنظام لا للتاجر |
| زبون جديد | دفع كامل مسبق دائماً |
| خيار الدفع الكامل | يظهر بنفس تنسيق الخيار الأساسي إذا لم تكن الاستراتيجية "دفع كامل" أصلاً |
| رفض الزبون الدفع | يغلق الصفحة — لا إشعار للتاجر |
| انتهاء الرابط | 24 ساعة من الإنشاء |
| رابط الزبون | رابط واحد طوال العملية — حالة الصفحة تتغير |
| الباقي (عند العربون) | يُحجز من نفس البطاقة عند تأكيد الاستلام |
| تحرير المال — يدوي | تأكيد **مفرد** من الزبون فقط — لا سائق، الزبون يؤكد مباشرة بعد الدفع |
| تحرير المال — Webhook | تأكيد **مزدوج** إلزامي: Webhook السائق أولاً ثم تأكيد الزبون |
| الرفض | تكلفة التوصيل من الزبون — Risk Score ينخفض تلقائياً |
| المشكلة/الخلاف | Escrow يتجمد + تكلفة توصيل تُخصم + الباقي للزبون — الحل خارج وصّل |
| الـ 48 ساعة | تبدأ من لحظة الدفع — لا إشعار — تحرير تلقائي |
| إلغاء الطلب | متاح للتاجر قبل دفع الزبون فقط |
| Stats | 5: إجمالي + Escrow + إفراج + إرجاع % + عدد مرجعة |
| جدول الطلبات | رقم · اسم · هاتف · منتج · مبلغ · استراتيجية · Escrow · تاريخ · إجراء |
| Real-time | Dashboard يتحدث تلقائياً بدون refresh |

---

## الـ Actors

| Actor | الدور | نقطة التفاعل |
|-------|-------|--------------|
| **التاجر** | ينشئ الطلبات ويتابع المال | Dashboard وصّل (يدوي أو عبر Webhook) |
| **الزبون** | يدفع ويؤكد الاستلام | رابط واحد طوال العملية |
| **شركة التوصيل** *(اختياري)* | تُسلّم وتُبلّغ عن الحالة — فقط في حالة Webhook | Webhook ثنائي الاتجاه |

## حالات Escrow الكاملة

```
empty           → لم يدفع الزبون بعد
funded          → الزبون دفع — محجوز (48h تبدأ)
driver_confirmed → السائق أكد التسليم — ننتظر الزبون
released        → تأكيد مزدوج مكتمل — تم الإفراج ✅
auto_released   → انتهت 48h — تحرير تلقائي ✅
frozen          → الزبون أبلغ عن مشكلة ⚠️
refunded        → أُعيد للزبون (رفض أو مشكلة) ↩️
cancelled       → التاجر ألغى قبل الدفع ✗
expired         → انتهى الرابط (24h) بدون دفع ⏰
```

## السيناريو D — رفض الاستلام

السائق يضغط "رُفض الاستلام".

**وصّل (في الخلفية):**
```
← Webhook: { event: "shipment.refused" }

إذا دفع عربون (30₪):
  refund = 30 - تكلفة التوصيل (20₪) = 10₪
  Merchant.balance += 0
  Customer refund = 10₪

إذا دفع كامل (150₪):
  refund = 150 - تكلفة التوصيل (20₪) = 130₪
  Merchant.balance += 0
  Customer refund = 130₪

← EscrowAccount.status → "refunded"
← Order.status → "returned"
← Customer.risk_score ينخفض تلقائياً
```

## السيناريو F — انتهاء 48 ساعة تلقائياً

الزبون دفع ولم يرد خلال 48 ساعة.

**وصّل — Cron Job:**
```python
# يعمل كل 5 دقائق
for escrow in escrows where status in ("funded", "driver_confirmed"):
    if datetime.utcnow() >= escrow.auto_release_at:
        if escrow.order.strategy != "full":
            # يحجز الباقي من البطاقة
            charge_remaining(escrow.order)
        escrow.status = "auto_released"
        merchant.balance += escrow.merchant_share
        order.status = "completed"
```

**واجهة التاجر — Dashboard:**

└─────────────────────────────────────────┘
